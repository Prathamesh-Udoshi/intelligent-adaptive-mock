Evolutionary Mock
User Guide

Everything you need to connect your API, navigate the dashboard, and get the most out of your intelligent Digital Twin.

Overview

Evolutionary Mock is a self-learning API proxy. It sits between your frontend and backend, observing every request and response. Over time it builds a precise behavioral model of your API — latency distributions, schema shapes, status code probabilities — and uses that model to serve high-fidelity mocks even when your real backend is offline.

👁️

Passthrough Proxy

In Ghost (Proxy) mode, all traffic passes through transparently while the AI silently observes and learns.

🚀

Mock Serving

In Rocket (Mock) mode, the platform serves AI-generated responses from its learned model — no real backend needed.

🚨

Contract Drift Detection

Continuously monitors the API structure. Any field changes, type shifts, or removals trigger a classified alert.

🩺

Health Scoring

Each endpoint receives a real-time health score (0–100) based on latency, error rate, and response size anomalies.

Quickstart

Get up and running in under two minutes.

  1. Open the Landing Page Navigate to your deployment URL (e.g. https://your-app.onrender.com). You'll see the setup panel.
  2. Enter Your Backend URL Paste the base URL of the API you want to observe into the Connect Your API box and click CONNECT. Example: https://api.example.com
  3. Update Your Frontend Replace your original API base URL with the Proxy URL shown on the landing page (which is just the root of this deployment). All requests now flow through the platform.
  4. Send Traffic Use your frontend app as normal. Every request is observed and learned. Check the Control Deck to see the live stream.
  5. Explore & Export After a few requests, visit the Explorer to inspect learned schemas, health scores, and drift alerts. Export TypeScript types from the same page.
💡 No real backend? You can use the free test API https://httpbin.org as your target to explore the platform before connecting your own service.

Proxy Setup

The platform acts as a transparent HTTP proxy. Your frontend talks to it as if it were the real backend — the platform forwards requests, learns from responses, and returns them unchanged.

Changing the Proxy Target

You can update the target URL at any time without restarting the server — either from the landing page setup panel or via the API directly:

# Using curl: curl -X POST https://your-app.onrender.com/admin/target \ -H "Content-Type: application/json" \ -d '{"target_url": "https://api.example.com"}'

Updating Your Frontend

Make a one-line change in your frontend code. Replace your API base URL with the platform's URL:

// Before const API_BASE = "https://api.example.com"; // After — point to the Evolutionary Mock platform const API_BASE = "https://your-app.onrender.com";
ℹ️ The proxy preserves all headers, query parameters, request bodies, and HTTP methods. Authenticated APIs work normally — the platform passes credentials through untouched.

CORS Considerations

If your frontend is running on a different origin (e.g., localhost:3000) and the platform is deployed remotely, ensure the platform's CORS policy allows your frontend's origin. The platform already allows all origins by default in development mode.

Using with Localhost & ngrok

The platform is deployed on a remote server (e.g. Render), which means it cannot reach your localhost directly. To proxy a locally running backend, you need to expose it to the internet using a tunneling tool. ngrok is the recommended solution — it's free, takes 30 seconds to set up, and gives you a public HTTPS URL for your local server.

⚠️ The platform runs on a remote server. Addresses like http://localhost:8000 or http://127.0.0.1 refer to that remote server's localhost — not yours. You must use a tunnel.

Step 1 — Install ngrok

# macOS (Homebrew) brew install ngrok # Windows (Chocolatey) choco install ngrok # Or download directly from ngrok.com/download

Step 2 — Sign Up & Set Auth Token (Free)

Create a free ngrok account at ngrok.com. Then authenticate your local CLI (one-time setup):

ngrok config add-authtoken YOUR_AUTHTOKEN_HERE

Step 3 — Expose Your Local Backend

Start your local API server, then run ngrok to tunnel it. Replace 8000 with your actual port:

# In terminal 1: start your local API (example: FastAPI) uvicorn main:app --port 8000 # In terminal 2: expose port 8000 to the internet ngrok http 8000

ngrok will output a public URL that looks like this:

Forwarding https://a1b2-123-45-67-89.ngrok-free.app → http://localhost:8000

Step 4 — Connect the Platform to Your Tunnel

Copy the https:// ngrok URL and paste it into the Connect Your API box on the landing page, then click CONNECT.

# Your target URL should be the ngrok HTTPS URL, NOT localhost target_url = "https://a1b2-123-45-67-89.ngrok-free.app" # ✅ Correct target_url = "http://localhost:8000" # ❌ Won't work from remote
✅ That's it! The platform will now proxy all traffic through ngrok to your local backend. You'll see requests appearing in real-time in the Control Deck.

ngrok Free Tier Limits

Feature Free Tier Notes
Tunnels 1 simultaneous Sufficient for development
Requests/min 40 RPM More than enough for learning
Session duration 2 hours (without auth token) Unlimited with free account
HTTPS ✅ Included Always use the https:// URL
Custom domains ❌ Not included URL changes each session on free tier
⚠️ On the free tier, your ngrok URL changes each time you restart ngrok. Remember to update the target URL in the platform's setup panel whenever this happens.

Alternative: Cloudflare Tunnel (Free, Persistent)

For a stable public URL that doesn't change, Cloudflare Tunnel (formerly Argo Tunnel) is an excellent free alternative with a persistent subdomain:

# Install cloudflared brew install cloudflared # Start a quick tunnel (no login required for testing) cloudflared tunnel --url http://localhost:8000

Control Deck

The Control Deck at /admin/dashboard is your main operational interface. It provides real-time visibility into every request flowing through the platform.

Live Stream

The right panel shows a real-time table of every request processed. Updates arrive via WebSocket — there is no polling delay. Each row shows:

Column Description
Time Timestamp of the request (localized to your browser timezone)
Type AI for Mock responses, REAL for proxied traffic
Request HTTP method and path pattern
Status HTTP response code (green = 2xx, red = errors)
Latency Round-trip time in milliseconds
Health 🟢 Healthy · 🟡 Degraded · 🔴 Critical
Drift 🚨 icon appears if a contract drift was detected on this response

Real-time Latency Chart

The chart above the log stream plots latency over time, with separate lines for Mock (purple) and Real/Proxy (green) traffic. It maintains the last 30 data points.

AI Brain Inspector

Select any endpoint from the dropdown in the left panel to inspect the schemas the platform has learned. Switch between:

  • Inbound (Request) — the JSON body structure sent to this endpoint
  • Outbound (Response) — the JSON response shape the platform uses to generate mocks

You can edit these schemas directly and click Update Knowledge to override what the platform has learned — useful for correcting edge cases or seeding schemas before traffic arrives.

Test Console

Use the Rocket Fire (Mock) and Ghost Fire (Proxy) buttons to manually trigger requests to a selected endpoint — useful for verifying mock fidelity without needing your frontend.

Learning & Mock Modes

Ghost Mode (Proxy)

The default operating mode. Every request is forwarded to your real backend and the response is returned normally. In parallel, the platform learns:

  • Request and response JSON schemas (with nested field detection)
  • Latency distribution (mean, standard deviation)
  • Status code probability distribution
  • Response body size distribution

Rocket Mode (Mock)

Switch to mock mode via the Control Deck when you want to serve AI-generated responses instead of hitting your real backend. Useful when:

  • Your backend is unavailable or under maintenance
  • You want deterministic responses for frontend testing
  • You're testing against a specific response shape
ℹ️ You need at least a few real requests before Mock Mode produces meaningful responses. The platform needs traffic to learn from before it can generate accurate mocks.

Global Learning Toggle

The Global Learning toggle in the Control Deck pauses schema learning while allowing traffic to continue flowing through. Use this when you want the proxy to stop updating its model — for example, during a known degraded state of your backend.

Chaos Engine

The Chaos Engine injects artificial failure scenarios into your traffic to test how resilient your frontend is. There are two controls:

Failure Rate Slider

Sets the probability (0–100%) that any given request returns an error instead of a real response. At 50%, roughly half of all requests will fail with an error code.

Incident Profiles

Pre-configured chaos scenarios that combine failure rate, latency injection, and error types. Available profiles include:

Profile Behavior Use Case
Normal No chaos applied Standard traffic observation
Degraded Moderate latency added, some errors Simulate a struggling backend
Outage High failure rate, long delays Test frontend offline handling / fallback UI
Flaky Random intermittent failures Test retry logic and error recovery
🔴 Chaos settings apply globally across all endpoints. Remember to reset the profile to Normal before resuming normal development.

API Explorer

The Explorer at /admin/explorer gives you a structured view of every endpoint the platform has discovered. It updates automatically (every 30 seconds with Live Sync enabled) and supports search and pagination for large APIs.

Endpoint Cards

Each card shows:

  • Method badge — GET, POST, PUT, PATCH, DELETE
  • Path pattern — dynamic segments are normalized (e.g. /users/{id})
  • Health score — 0–100 with colour coding
  • Average latency and error rate
  • Status code distribution (e.g. 200 (87%), 404 (13%))
  • Learned schemas — request and response bodies side by side
  • Active drift alerts with AI narration (if any)

Search

Use the search bar to filter endpoints by path pattern. The filter is applied server-side and supports partial matches (e.g., search user to find /users, /users/{id}, /admin/users, etc.).

Path Intelligence

The platform automatically groups requests with dynamic path segments. For example, /users/123, /users/456, and /users/abc are all grouped under the single pattern /users/{id}.

Contract Drift Alerts

Contract drift occurs when your backend API changes its response shape without a corresponding frontend update — a leading cause of silent runtime bugs. The platform detects this automatically by continuously comparing live responses against the learned schema baseline.

Drift Severity Levels

Each detected change is classified into one of three severity tiers:

BREAKING Required fields removed, type changed (e.g. string → number), nested object restructured
WARNING Optional fields removed, null values introduced where previously absent
INFO New optional fields added, response body expanded

AI Narrator Report

Each drift alert includes a human-readable AI Narrator Report that describes exactly what changed, in plain language — what was removed, what type shifted, and what the likely impact is. Toggle to Technical Details view to see the raw field paths and diff.

Resolving an Alert

Once you've reviewed a drift alert and updated your frontend accordingly, click Resolve on the alert card in the Explorer. This marks the alert as resolved and updates the schema baseline to the new structure, so future comparisons use the current shape.

ℹ️ Only the most recent unresolved alert per endpoint is shown in the Explorer. Historical alerts are retained in the database for audit purposes.

AI & ML Neuro-Cognitive Engine

Evolutionary Mock features a hybrid AI architecture designed to catch complex behavioral regressions that traditional monitoring tools miss. Our Neuro-Cognitive Engine uses two distinct layers of analysis:

📊

Statistical Engine (Welford)

Provides fast, real-time detection of latency spikes and error rate jumps using adaptive standard deviations. It creates a baseline for every unique endpoint pattern.

🧠

Neural Engine (LSTM)

A deep-learning Autoencoder that understands the relationship between timing, size, and payload patterns. It flags anomalies where metrics look "normal" but the "signature" is wrong.

Real-Time Training Transparency

We believe AI should not be a "Black Box." The AI Cognitive Status panel on the Control Deck shows you exactly what's happening under the hood:

  • Initial Accumulation: Tracking the first 30 healthy requests needed to bootstrap the engines.
  • Training States: Visual feedback when the platform is actively training neural weights in the background.
  • Active Engine Badges: Identifies which specific AI engine (Welford or LSTM) flagged a particular transaction.

Health Monitor

Every endpoint is continuously scored on a 0–100 health scale using a sliding window of recent observations. The score reflects how far the current behaviour deviates from the learned baseline.

Health Score Bands

Score Status Meaning
80 – 100 🟢 Healthy All metrics within normal learned baselines
50 – 79 🟡 Degraded One or more anomalies detected — review recommended
0 – 49 🔴 Critical Multiple severe anomalies — immediate attention required

What Is Monitored

The platform monitors five primary signals per endpoint using a Dual-Engine (Statistical + Neural) architecture:

  • Latency — spikes beyond a dynamic threshold from the learned mean are flagged as statistical anomalies
  • Neural Signatures — a deep learning LSTM Autoencoder detects pattern shifts in the overall traffic "DNA"
  • Error Rate — jumps above 3× the baseline error rate trigger a warning
  • Response Size — changes beyond 3× the average response size are flagged
  • Contract Drift — structural schema changes detected by the regression watchdog

Adaptive AI Thresholding

The platform doesn't use a single "one-size-fits-all" threshold for latency. Instead, it uses the Coefficient of Variation (CV) to adjust its sensitivity per endpoint:

  • Stable Endpoints: If an endpoint is remarkably consistent (e.g. 10ms ± 1ms), the AI becomes more sensitive, flagging even a small jump as an anomaly.
  • Volatile Endpoints: For jittery endpoints like LLM generations or Search APIs, the AI automatically expands its patience, reducing false alerts while still catching massive failures.

You can see the current AI Sensitivity (σ) in the Control Deck — it represents the dynamic Z-score threshold the system has calculated for your current project.

Platform Health Banner

The Control Deck shows an aggregated Platform Health banner at the top — the lowest health score across all monitored endpoints. If any endpoint is critical, the banner turns red to draw your attention immediately.

Type Exports

The Explorer page allows you to export your learned API schemas as ready-to-use type definitions — eliminating the need to hand-write types for existing endpoints.

Supported Formats

🟦

TypeScript

Generates interface definitions for every learned endpoint response, ready to paste into your frontend project.

🐍

Generates Python BaseModel classes for use in FastAPI, data validation pipelines, or test fixtures.

📄

JSON Schema

Generates standard JSON Schema definitions, compatible with any language or validation library that supports the spec.

📋

Copy & Download

One-click copy to clipboard or download as a file (.ts, .py, .json) for immediate use.

Open the Export Client Types panel at the top of the Explorer page, select a format, and click generate. The output reflects every endpoint currently in the platform's knowledge base.

OpenAPI / Swagger

The platform generates a live OpenAPI 3.0 specification from its learned traffic, and exposes an interactive Swagger UI you can use to browse and test endpoints.

  • Swagger UI — /admin/docs
  • Raw OpenAPI JSON — /admin/export-openapi

The generated spec includes:

  • All discovered endpoint paths and their HTTP methods
  • Path parameter detection (e.g. {id}, {slug})
  • Example request and response bodies from real traffic
  • Status code distributions with probabilities as descriptions
ℹ️ The spec is generated dynamically — it reflects the latest learned state each time it's fetched. Import the URL directly into Postman, Insomnia, or any OpenAPI-compatible tool.

Manual Endpoint Definition

You can seed the platform with endpoints before any real traffic arrives. This is useful when your backend isn't ready yet but you want to develop your frontend against a contract.

Using the Landing Page Form

On the landing page, scroll to Define Endpoints Manually. Select an HTTP method, enter the path (using {param} notation for dynamic segments), and paste a sample JSON response body. Click + Add Endpoint.

Via the API

curl -X POST https://your-app.onrender.com/admin/endpoints/manual \ -H "Content-Type: application/json" \ -d '{ "method": "GET", "path": "/users/{id}", "status_code": 200, "response_body": { "id": 1, "name": "Jane Doe", "email": "jane@example.com", "role": "admin" } }'

Once defined, the endpoint immediately appears in the Explorer and is available for mock serving in Rocket Mode.

Schema Editor

The AI Brain Inspector in the Control Deck is a live schema editor. You can directly modify what the platform has learned for any endpoint — useful for correcting AI inferences or handling edge cases.

  1. Select an Endpoint Use the dropdown in the left panel of the Control Deck to select the endpoint you want to edit.
  2. Choose Schema Type Toggle between Inbound (Request) and Outbound (Response) using the tab selector.
  3. Edit the JSON The textarea loads the current learned schema as editable JSON. Modify fields, add properties, or restructure as needed.
  4. Save Click Update Knowledge. The platform immediately uses the updated schema for future mock responses.
⚠️ Schema edits are overwritten if new real traffic arrives while Global Learning is enabled. Disable learning before editing if you want your manual changes to persist.
